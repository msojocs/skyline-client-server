一、致命性能瓶颈（Critical）
1. 每个 DOM 操作都是同步 TCP 往返 — 无批处理机制
位置: base_client.cc:12-28, node.cc 中所有方法
问题: 构建一个典型页面可能需要数百次 DOM 操作（createElement → setStyleScope → addClass → setStyle → appendChild → ...），每一次都是：
Napi→JSON序列化 → TCP发送 → 等待响应 → JSON反序列化 → TCP接收 → JSON解析 → 返回
一次 TCP 往返延迟约 0.1-1ms（localhost），数百次操作意味着 100-500ms 纯通信延迟。
优化方案 — 操作批处理（Batching）:
// 新增 BatchContext，客户端积攒操作后一次性发送
class BatchContext {
    std::vector<nlohmann::json> operations;
public:
    void add(nlohmann::json&& op) { operations.push_back(std::move(op)); }
    
    // 一次性发送所有操作，获得所有结果
    std::vector<nlohmann::json> flush() {
        nlohmann::json batch = {
            {"type", "batch"},
            {"operations", std::move(operations)}
        };
        operations.clear();
        return ClientAction::sendBatchSync(batch);
    }
};
对于不需要返回值的操作（setStyle, addClass, setAttribute, setStyleScope, appendChild 等），可以直接入队延迟发送。对于需要返回值的操作（createElement, getBoundingClientRect），则先 flush 积攒的操作再执行。
预估收益: 减少 70-90% 的 TCP 往返次数，页面构建时间可从数百毫秒降到数十毫秒。
2. JSON 文本序列化 — 极其低效的协议格式
位置: convert.cc:83-144（序列化）, convert.cc:146-216（反序列化）, server_action.cc:191（服务端二次解析）
问题:
- 每条消息都做 nlohmann::json → .dump() → string → parse() 全文本序列化
- 数字 12345 在 JSON 中占 5 字节，在二进制中只需 4 字节
- 服务端 sendMessageSync (server_action.cc:191) 收到已经是字符串的消息，还要 parse 一次 JSON 仅仅为了注入 id 字段，然后再 dump 回字符串
优化方案 — 二进制协议:
选项 A（渐进式）：使用 MessagePack（nlohmann::json 原生支持）：
// 序列化：只需改两行
auto msgpack = nlohmann::json::to_msgpack(data);
// 反序列化
auto json = nlohmann::json::from_msgpack(buffer);
MessagePack 比 JSON 快 2-5x，体积小 30-50%。
选项 B（激进式）：使用 FlatBuffers，零拷贝反序列化。
选项 C（最小改动）：修复服务端的二次解析问题，将 id 通过协议头传递而非注入 JSON：
+---4 bytes---+---8 bytes---+---N bytes---+
| msg length  |  request id |  payload    |
+-------------+-------------+-------------+
预估收益: 序列化/反序列化性能提升 2-5x。
3. Buffer/ArrayBuffer 逐字节 JSON 序列化
位置: convert.cc:117-132
// 当前实现：1MB buffer → 生成约 5MB 的 JSON 数组 "[0,1,2,255,128,...]"
for (uint32_t i = 0; i < byteLength; i++) {
    jsonArr[i] = buffer.Data()[i];
}
问题: 一个 1MB 的 Buffer 会变成约 5MB 的 JSON 字符串（每个字节变成 1-3 位数字 + 逗号）。解析端还要逐元素读回。
优化方案:
- 方案 A：Base64 编码（体积 1.33x，但解析极快）
- 方案 B：扩展共享内存机制到所有二进制数据（目前只有 notifyHttpRequestComplete 用了共享内存）
- 方案 C：如果改用 MessagePack，二进制数据原生支持 bin 类型
预估收益: 二进制数据传输性能提升 10-100x。

二、高优先级性能问题（High）
4. 忙等待轮询 — 浪费 CPU 且增加延迟
位置: client_action.cc:208-265
while (true) {
    // ... 处理回调队列 ...
    if (futureObj.wait_for(std::chrono::milliseconds(2)) == std::future_status::ready) {
        break;
    }
    std::unique_lock<std::mutex> lock(socketRequestMutex);
    socketEventCv.wait_for(lock, std::chrono::milliseconds(2));
}
问题: 
- 2ms 的轮询间隔意味着平均额外增加 1ms 延迟
- 同时做了 future::wait_for 和 condition_variable::wait_for，双重等待
- 在等待期间还要持有 socketRequestMutex，可能阻塞接收线程
优化方案:
// 方案：只使用 condition_variable，response 到达时 notify
while (true) {
    // 先处理回调队列（无锁检查）
    processCallbackQueue();
    
    std::unique_lock<std::mutex> lock(socketRequestMutex);
    // 等待直到：响应到达 OR 有新回调 OR 超时
    socketEventCv.wait_for(lock, std::chrono::milliseconds(100), [&] {
        return futureObj.wait_for(std::chrono::microseconds(0)) == std::future_status::ready
            || hasCallbacks();
    });
    
    if (futureObj.wait_for(std::chrono::microseconds(0)) == std::future_status::ready) {
        break;
    }
    // 检查超时...
}
预估收益: 减少平均 1ms 延迟/次调用，降低 CPU 空转。
5. 服务端 getInstanceId 线性扫描 O(n)
位置: object-manage.ts:47-53
getInstanceId: (instance: any) => {
    for (const [key, value] of instanceMap.entries()) {
        if (value === instance) return key  // O(n) 线性扫描!
    }
    return null
}
问题: 每次需要查找一个实例的 ID 时（在 hookCallbackArgument 和 hookResult 中频繁调用），都要遍历整个 instanceMap。随着实例数增长，性能线性下降。
优化方案:
const instanceMap = new Map<number, any>();
const reverseMap = new WeakMap<object, number>();  // 反向索引
setInstance: (instance: any) => {
    const id = instanceId++;
    instanceMap.set(id, instance);
    if (typeof instance === 'object' && instance !== null) {
        reverseMap.set(instance, id);
    }
    return id;
},
getInstanceId: (instance: any) => {
    return reverseMap.get(instance) ?? null;  // O(1)
}
预估收益: 实例查找从 O(n) 降到 O(1)。
6. 回调注册永不释放 — 内存泄漏
位置: convert.cc:105-108（客户端回调），object-manage.ts:40-42（服务端实例）
// 客户端：每个函数参数都注册一个新的 callback + ThreadSafeFunction，永不删除
callback[cId] = {
    std::make_shared<Napi::FunctionReference>(Napi::Persistent(func)),
    Napi::ThreadSafeFunction::New(env, func, "Callback", 0, 1)
};
// 服务端：实例只增不减
setInstance: (instance: any) => {
    const id = instanceId++
    instanceMap.set(id, instance)  // 永不删除
    return id
}
同样，客户端的 instanceCache（convert.cc:27）也永不清理。
优化方案: 
- 实现引用计数或基于 release() 调用的清理机制
- 当 ShadowNode::release() 被调用时，同时清理客户端 instanceCache 中的对应条目
- 对回调使用弱引用或显式 unregister 机制
- 服务端在 removeInstance 时同时清理 reverseMap
预估收益: 防止长时间运行时的内存膨胀和查找退化。

三、中优先级优化（Medium）
7. matches() 本地优化未完成
位置: node.cc:98-137
Napi::Value ShadowNode::matches(const Napi::CallbackInfo &info) {
    // 快速路径：ID 匹配
    if (idView == this->id) return true;
    
    // 类名匹配已注释掉！
    // else if (last[0] == '.') { ... }
    
    // 回退到 TCP 往返
    return sendToServerSync(info, __func__);
}
优化方案: 完成本地类名匹配（被注释掉的代码），可以省去大量 matches() 的 TCP 往返：
if (!lastSelector.empty() && lastSelector[0] == '.') {
    std::string_view className = lastSelector.substr(1);
    for (const auto& cls : classList) {
        if (cls == className) {
            return Napi::Boolean::New(info.Env(), true);
        }
    }
    // 也可以用 unordered_set 替代 vector 实现 O(1) 查找
}
// 仅复杂选择器才回退到服务端
return sendToServerSync(info, __func__);
同时考虑将 classList 从 std::vector<std::string> 改为 std::unordered_set<std::string> 以实现 O(1) 查找。
预估收益: 消除大部分 matches() 调用的 TCP 往返。
8. 部分不需要返回值的操作可改为异步
位置: node.cc 中大量方法
以下操作语义上不需要等待服务端返回值：
- setStyle, addClass, setClass, clearClasses, removeClass
- setAttribute, setAttributes, removeAttribute
- setId, setStyleScope, setEventDefaultPrevented
- setListenerOption, setTouchEventNeedsLocalCoords
- markAsListItem, markAsRepaintBoundary, preventSnapshot
- addAnimatedStyle, clearAnimatedStyle, appendStyle, updateStyle
- associateComponent
这些可以改为 sendToServerAsync（fire-and-forget），不阻塞主线程：
Napi::Value ShadowNode::setStyle(const Napi::CallbackInfo &info) {
    return sendToServerAsync(info, __func__);  // 不等待返回
}
注意: 需要保证操作顺序性。可以使用序列号或者在需要返回值的操作前 flush 所有 pending 的异步操作。这与方案 1（批处理）可以结合使用。
预估收益: 大幅减少阻塞等待时间。
9. 消息处理中的冗余 JSON 解析
位置: 
- client_action.cc:33 — 接收线程解析 JSON
- client_action.cc:274 — 主线程再次解析同一消息
- server_action.cc:39 — 接收线程解析
- server_action.cc:251 — 主线程再次解析
消息被解析了两次：一次在接收线程（用于路由），一次在主线程（用于处理结果）。
优化方案: 接收线程只解析必要字段（id 和 type），传递原始字符串给主线程或用二进制协议头避免解析。
10. ThreadSafeFunction 开销
位置: convert.cc:107
每注册一个回调都创建一个 Napi::ThreadSafeFunction：
callback[cId] = {
    std::make_shared<Napi::FunctionReference>(Napi::Persistent(func)),
    Napi::ThreadSafeFunction::New(env, func, "Callback", 0, 1)  // 每个回调一个 TSFN
};
ThreadSafeFunction 是相对重的资源。考虑使用单个共享的 TSFN + 消息队列替代多个独立 TSFN。

四、低优先级优化（Low）
11. 考虑使用 Unix Domain Socket 替代 TCP
位置: client_socket.cc, server_socket.cc
当前使用 TCP localhost:3001。由于客户端运行在 Linux、服务端运行在 Wine，如果 Wine 支持 Unix Domain Socket，可以切换：
- UDS 比 TCP 少了协议栈开销（无 TCP 握手、校验和、拥塞控制）
- 延迟通常低 20-30%
不过需要确认 Wine 对 UDS 的支持情况。
12. 编译优化建议
位置: CMakeLists.txt:38-49
当前已有 -O3 -march=native -flto，建议额外添加：
# Profile-guided optimization
set(CMAKE_CXX_FLAGS_RELEASE "${CMAKE_CXX_FLAGS_RELEASE} -fprofile-generate")
# 运行测试后
set(CMAKE_CXX_FLAGS_RELEASE "${CMAKE_CXX_FLAGS_RELEASE} -fprofile-use")
# 更好的内联
set(CMAKE_CXX_FLAGS_RELEASE "${CMAKE_CXX_FLAGS_RELEASE} -finline-functions -funroll-loops")
13. 减少 nlohmann::json 临时对象
位置: client_action.cc:294-303, client_action.cc:307-316 等多处
nlohmann::json json {
    {"type", "dynamic"},
    {"action", action},
    {"data", {{"instanceId", instanceId}, {"params", args}}}
};
每次调用都创建完整的 JSON 对象树。考虑预分配或使用对象池。

五、优化优先级总结
优先级	方案	预估收益
P0	1. DOM操作批处理	减少70-90% TCP往返
P0	2. 二进制协议(MessagePack)	序列化性能提升2-5x
P0	3. Buffer不再逐字节序列化	二进制数据提升10-100x
P1	4. 消除忙等待轮询	每次调用减少~1ms延迟
P1	5. 反向索引O(1)查找	实例查找从O(n)→O(1)
P1	6. 回调/实例清理机制	防止内存泄漏
P2	7. 完成matches()本地优化	减少matches()往返
P2	8. 无返回值操作改异步	减少阻塞等待
P2	9. 消除冗余JSON解析	减少CPU开销
P2	10. 共享ThreadSafeFunction	减少资源占用
P3	11. Unix Domain Socket	延迟降低20-30%
P3	12. PGO编译优化	整体性能微提升
建议实施顺序: 先做 P1 中的低难度改进（方案 4、5），然后实施 P0 中的核心优化（方案 2→3→1），最后逐步补齐 P2 的优化。
方案 1（批处理）和方案 8（异步操作）可以结合为统一的 "命令流水线" 架构：不需要返回值的操作直接入队，需要返回值时先 flush 队列再执行同步调用。这是对当前架构改动最大但收益也最大的优化。
